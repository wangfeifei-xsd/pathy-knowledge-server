# 知识库维护约定（AGENTS）

本文档供编译、文档校验等自动化任务引用。请根据团队实际情况补充或修改各章节中的占位内容。

---

## 一、目录结构

```
knowledge-base/
├── raw/          # 原始素材：个人笔记、剪藏内容、未经整理的参考资料
├── wiki/         # 编译后的结构化条目：按主题分类的知识条目
└── schema/      # 规范文件：本文档及其他团队约定
```

> **说明**：raw 目录用于存放未经加工的原始内容，wiki 目录存放已编译的结构化知识，两者分离便于追溯来源与后续维护。

## 二、命名与格式规范

### 文件命名

- 使用 **小写字母 + 连字符**（kebab-case）或**下划线**（snake_case）的命名方式
- 文件名应简洁且具有描述性，例如：`user-authentication-flow.md`、`api_rate_limit.md`
- 常用约定文件名：`AGENTS.md`（本文档）、`README.md`、`CHANGELOG.md`

### 格式约定

- 内容采用 **Markdown** 语法编写
- 标题层级清晰，建议最多使用三级标题（H1 → H2 → H3）
- 代码块需标注语言类型，便于语法高亮

> 示例：团队内部是否要求特定前缀？是否禁止使用中文文件名？是否需要版本号？如有请补充。

## 三、禁止事项

为保障知识库的安全性、一致性与可维护性，以下行为应予以避免：

- **禁止在正文中存放明文密钥**、凭证、Token 等敏感信息（应使用环境变量或加密方式引用）
- **禁止删除他人的条目**，如需移除请与原作者确认或留档说明
- **禁止提交未经校验的内容**，至少完成基本的语法检查
- **禁止在公共页面泄露内部流程或机密信息**

### （在此补充团队specific的禁止规则）

> 示例：禁止上传大于 10MB 的文件、禁止使用外部未授权的图片链接等。

## 四、写作与链接风格

### 术语统一

- 首次出现的技术术语或专有名词应提供解释或链接
- 团队内部用语应保持一致的译名，建议在 wiki 根目录维护一份《术语表》

### 内部链接

- 建议使用相对路径链接，以便在不同部署环境下保持可用
- 链接文本应清晰表达目标内容，避免使用「点击这里」等模糊描述

### 文档结构

- 每个条目应包含明确的开头概述，便于快速了解内容定位
- 关键信息置于文档前部，细节与参考资料可附录或以折叠形式呈现

### （在此补充术语、内部链接格式等）

> 示例：链接样式 `[概念说明](./glossary.md#概念名)`，或约定使用 `[[双链]]` 语法。

## 五、维护建议

- **定期巡检**：建议每季度对 wiki 目录进行一次内容审查，清理已过时或重复的条目
- **版本记录**：重要条目更新时应保留变更记录，可通过 Git 历史或 CHANGELOG 追踪
- **协作约定**：新增或大幅修改条目时，建议通过 Pull Request 流程进行评审
